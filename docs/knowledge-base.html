Developer Knowledge Base

A comprehensive guide on building with, configuring, and coding alongside the Nexus Prime OS layer.

1. Introduction

Nexus Prime is designed to be completely invisible to the end user, but completely exposed to the AI Coding Agent. By running as an MCP server in the background, agents (like AntiGravity, Claude Desktop, or custom scripts) can invoke its tools natively.

Mental Model

Treat Nexus Prime as a co-processor. If you (the agent) are unsure about history, you ping the Hippocampus. If you are reading heavy logs, you route them through the Token Optimizer. If you are refactoring multiple interrelated files, you spawn Phantom Workers.

2. Nexus Prime Language Guideline (for Agents)

If you are an AI coding agent connected to this workspace, you are operating within the Super Intellect Stack. Your prompts, code exploration, and problem-solving strategies must follow the Core Directives.

The Agent Language Directive

When coding in a Nexus Prime environment, you must adhere to the following principles:

  • Protocol Over Assumption: Never guess what happened prior to your birth (session start). You. MUST. Call. nexus_recall_memory on first boot.
  • Token Frugality: If a file is over 500 lines, do not read it raw. Pass it through nexus_optimize_tokens. Listen to the routing (Read, Outline, or Skip). Ignorance of routing is a Guardrail violation.
  • Fission on Insight: When you decipher a complex architecture, figure out a bug, or make an architectural decision, immediately invoke nexus_store_memory. Write it like a Zettelkasten note (concise, searchable, heavily tagged).
  • Pre-Flight Checks: Before deleting files, rewriting core modules, or altering >3 files simultaneously, run nexus_ghost_pass to detect rippling dependencies via the AST.
  • Swarm Delegation: When the `Ghost Pass` reveals heavily coupled dependencies and you aren't sure which path is correct, DO NOT guess. Use nexus_spawn_workers to split the worktree and run experiments in parallel.

3. Reading Session Telemetry

Nexus Prime automatically formats its log outputs back to the MCP Client using Grain primitives. Every successful MCP action appends a state header to the response:

<state type="telemetry">
─── 📡 Nexus Prime (12s) ───
3 calls │ 20.3k tokens saved │ 2 stored │ 5 recalled
</state>

As an agent, you do not need to parse this. It is designed to be injected into your prompt context so that you are passively aware of your savings and memory accumulation throughout the session.

4. API Reference

Memory Operations

Tool Name Required Params Description
nexus_recall_memory query (str), k (num) Performs a Vector/TF-IDF similarity search across the SQLite Cortex database. Retrieves historical context.
nexus_store_memory content (str), priority (num), tags (arr) Persists an insight to the Cortex. Priority > 0.8 keeps the memory active in the Hippocampus cache.
nexus_memory_stats None Returns Zettelkasten metrics: total nodes, dominant tags, and cache tier capacities.

Intelligence Ops

Tool Name Required Params Description
nexus_optimize_tokens task (str), files (arr) Evaluates codebase relevancy relative to your task. Returns a strict reading plan (Read, Outline, Skip).
nexus_ghost_pass goal (str), files (arr) Performs a read-only AST simulation on target files to spot syntax risks and breaking changes before they happen.
nexus_mindkit_check action, filesToModify Strict policy evaluation. Returns PASS/FAIL based on Token Budgets and the Destructive Guard filters.

Parallel Workers

Tool Name Required Params Description
nexus_spawn_workers goal (str), files (arr) Spawns headless Phantom Workers into isolated `git worktrees`. Evaluates differing solutions via Byzantine Merge Oracle.
nexus_audit_evolution None Reads the SQLite history for "hotspot" files that break frequently. Pushes these evolution candidates back to MindKit sync.

5. MindKit Guardrails & Sync

Nexus Prime acts as the enforcement layer for the policies defined by the sir-ad/mindkit repository.

Destructive File Protection

Any agent attempting to rewrite configurations spanning more than 10 files simultaneously will automatically trigger the BULK_FILE_GUARD. The MCP server will respond with a warning exception requesting a fallback to a nexus_ghost_pass.

Bidirectional MindKit Sync

When the nexus_audit_evolution tool runs, Nexus Prime pushes analytical telemetry directly back to the MindKit repository via the GitHub REST API. This forms a continuous feedback loop where MindKit's rules get smarter based on the bugs you encounter in your local environment.
